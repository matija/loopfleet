2026-07-06 — M0: SQLite store with migrations for the data model
- Added rusqlite 0.32 (feature "bundled" — vendors SQLite, no system dep) to the
  workspace deps and the loopfleet-store crate.
- crates/store/src/migrations/0001_init.sql: full initial schema matching the PRD
  data model — projects, plans, tasks (PK = plan_id + anchor normalized_text,
  line_hint as tiebreaker), runs (Model B: plan_id + task_anchor FK to tasks),
  iterations, sessions (M5 schema defined now), events (single append-only log,
  AUTOINCREMENT seq). Derived TaskStatus is intentionally NOT stored. Indexes on
  events(owner,seq), runs(plan_id,task_anchor), plans(project_id).
- crates/store/src/lib.rs: migration runner. MIGRATIONS is an ordered
  (name, include_str! sql) list; migrate() creates schema_migrations, applies
  each pending migration in its own BEGIN/COMMIT with ROLLBACK on error, skips
  already-applied ones (idempotent). open(path) opens a connection, enables
  foreign_keys pragma, and migrates. ":memory:" supported for tests.
- Tests (4, all passing): full schema present after open, migrate idempotent,
  foreign keys enforced (plan -> missing project rejected), events.seq
  autoincrements.

Verification:
- CARGO_HOME=/tmp/loopfleet-cargo cargo test -p loopfleet-store — 4 passed.
- cargo clippy -p loopfleet-store — clean.
- cargo build --workspace — Finished (store + Tauri app compile).

Design notes:
- Migrations are append-only and keyed by name; never edit or reorder a shipped
  entry. Next migration is 0002_*.sql appended to MIGRATIONS.
- foreign_keys pragma is per-connection, so open() sets it every time; migrate()
  alone does not (callers using migrate() directly must set the pragma).

2026-07-06 — M1: Normalized event enum (adapter-sourced vs app-sourced lanes)
- crates/core/src/event.rs: NormalizedEvent, the single vocabulary every
  downstream consumer speaks. Variants match the PRD exactly: adapter-sourced
  lane (TurnStarted, AssistantText { text }, Reasoning { text }, ToolCall
  { call_id, name, input_excerpt }, ToolResult { call_id, ok, output_excerpt },
  CommandRun { cmd, exit: Option<i32> }, TurnCompleted { usage }, NeedsApproval,
  Failed { reason }, Ended) and the app-sourced lane (FileChanged { path }).
  ToolCall/ToolResult correlate by call_id (a shared String field on both).
- Lane enum { Adapter, App } + NormalizedEvent::lane(): only FileChanged is
  App; everything else is Adapter. The lane distinction is expressed as a method
  rather than splitting the type, so downstream still consumes one enum.
- Usage { input_tokens, output_tokens }: u64, Default, unreported counts = 0.
- Serde: internally tagged `#[serde(tag = "kind", rename_all = "snake_case")]`,
  so the store's normalized_event_json column holds e.g.
  {"kind":"tool_call","call_id":"…",…} and unit variants {"kind":"ended"}.
  All variants are unit or struct (no tuple variants), which internal tagging
  requires.
- NeedsApproval kept fieldless per PRD (only fires in M5 interactive sessions;
  headless never surfaces it — the sandbox is the boundary). No speculative
  fields added; revisit if M5 needs correlation.
- Placement: core, per core/lib.rs's own doc comment ("the normalized event
  enum" lands here). Re-exported from lib.rs (Lane, NormalizedEvent, Usage).
  store stays agnostic — it persists opaque JSON text, not the Rust type.
- Added serde dep to core, serde_json to core dev-deps + workspace deps
  (adapters will need serde_json for their stream mapping in the next M1 tasks).
- Tests (4, all passing): round_trips_every_variant (every variant survives
  JSON round-trip), kind_tag_shape_is_stable (the snake_case discriminator),
  tool_call_and_result_correlate_by_call_id, lanes_classify_correctly.
