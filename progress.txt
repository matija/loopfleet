2026-07-06 — M1: Event-log writer (single-writer SQLite via a bounded channel)
- crates/store/src/event_log.rs: EventLog — the one place rows are appended to
  the `events` table. spawn(conn, capacity) takes ownership of the Connection and
  drains a bounded tokio mpsc of LogEntry on a tokio::task::spawn_blocking loop
  (rx.blocking_recv()), inserting each row. SQLite is single-writer, so funneling
  every producer through one writer means concurrent runs never contend for the
  write lock. The bounded channel IS the backpressure: a slow/bursty writer
  stalls producers at send().await instead of growing an unbounded buffer
  (matches the RunHandle mpsc(64) contract the adapters already use).
  - spawn_blocking (not a plain async task): keeps the synchronous rusqlite calls
    off the async runtime's worker threads; the writer returns
    rusqlite::Result<usize> (events written) so a write error stops the drain and
    surfaces on shutdown().await.
  - sender() -> clonable Sender (re-exported as EventSender): one clone per
    producer — a per-run adapter-forwarding task (M3) plus the app's worktree
    watcher (M2/M3). Writer stops when the last sender drops. shutdown() drops
    the log's own sender, awaits the writer, returns the count.
  - KEY BOUNDARY: the store stays AGNOSTIC to the event type. LogEntry carries
    { owner_id, event_json: String } — the already-encoded {"kind":…} JSON that
    NormalizedEvent serializes to — NOT the core enum. This is deliberate: core
    depends on store (register_project), so store depending on core would be a
    build cycle (cargo rejected it), and the enum's own design note says "store
    persists opaque JSON text, not the Rust type." Producers serialize before
    sending; consumers (load_events) get the JSON back and deserialize. So this
    task touched zero core code.
  - FileChanged (app-sourced lane) enters through the SAME Sender as
    adapter-sourced events — the channel is the merge point and imposes the
    ordering. Adapters never emit FileChanged; the app pushes it. The fs/git
    watcher that generates it needs worktrees, so it lands with M2/M3; M1
    establishes only that the app lane is a first-class writer input (proven by
    a test that mixes a TurnStarted from an "adapter" sender and a FileChanged
    from an "app" sender into one owner's log).
  - load_events(conn, owner_id) -> Vec<StoredEvent{seq,event_json,ts}>: reads a
    run/session's events back in append (seq) order via a separate connection.
    Used by the M4 timeline and by the tests here. Reads stay off the writer.
  - now_millis(): events.ts is unix millis, stamped at write time.

  (rt-multi-thread, macros) + tempfile "3" for the file-backed round-trip tests.

Design notes:
- The writer is intentionally event-type-agnostic: serialization is a boundary
  concern that belongs above the store, at the point that owns NormalizedEvent
  (the M3 supervisor's forwarding task). Keeping it out of store preserves the
  core→store dependency direction and keeps the store a pure persistence layer.
- spawn_blocking + blocking_recv is the single-writer thread model: one owned
  Connection, one drain loop, no locks, no async-blocking-on-sync footgun.
